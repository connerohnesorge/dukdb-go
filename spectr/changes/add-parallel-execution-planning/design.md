# Parallel Execution Planning - Design Details

## Implementation Details

### Pipeline Analysis

Pipeline analysis identifies boundaries where execution can be parallelized:

```
Blocking Operators (pipeline breakers):
- Sort
- Aggregate (with GROUP BY)
- Window Functions
- Distinct
- Limit (with ties or window functions)

Non-blocking Operators (streaming):
- Filter
- Project
- Unnest
- Lateral Join

Pipeline Chain Identification:
- Traverse plan tree top-down
- Build chains of streaming operators
- Mark blocking operators as pipeline boundaries
- Identify independent pipeline stages
```

**Key Structures**:
- `PipelineInfo`: Stores pipeline boundaries and stage information
- `OperatorType.PipelineProperty`: Classifies each operator as blocking or streaming

### Parallel Plan Generation

Parallel plans are generated by inserting Exchange operators and parallelizing compatible operators:

```
Sequential Plan:
TableScan(t1) → Filter(x=1) → HashJoin → Aggregate(SUM)

Parallel Plan:
[Worker 0] TableScan(morsel_0) → Filter(x=1) ──┐
[Worker 1] TableScan(morsel_1) → Filter(x=1) ──┤
[Worker 2] TableScan(morsel_2) → Filter(x=1) ──┤
                                                 v
                          Exchange(Gather) → Repartition(on join_key)
                                 │
                    ┌────────────┼────────────┐
            [Worker 0] HashJoin ─┤   [Worker 1] HashJoin ─┤   [Worker 2] HashJoin
                    │            │            │
                    └────────────┼────────────┘
                                 v
                          Exchange(Gather) → PartialAggregate
                                                 │
                                            GlobalAggregate
```

**Generation Strategy**:

1. **For Table Scans**: Partition by row groups, create one scan operator per worker
2. **For Joins**:
   - Insert repartition exchange on join key
   - Each worker hashes and partitions its build table
   - Workers execute hash join on their partition
   - Results gathered at top
3. **For Aggregations**:
   - Insert repartition exchange before aggregation
   - Each worker computes local aggregate
   - Final gather and merge aggregates
4. **For Sorts**:
   - Insert sample sort exchange
   - Each worker sorts locally
   - K-way merge at end

### Exchange Operators

Exchange operators manage data movement between parallel stages:

```go
// Exchange operator types
type ExchangeType int

const (
    ExchangeGather      // Collect from all workers into single output
    ExchangeRepartition // Redistribute by hash on column
    ExchangeBroadcast   // Replicate to all workers
    ExchangeRoundRobin  // Distribute in round-robin order
)

type PhysicalExchange struct {
    Type        ExchangeType
    PartitionBy []Expr  // For repartition
    PartitionCount int   // Target partition count
}
```

### Cost Model Extension

Sequential cost model is extended to account for parallelism:

```
ParallelCost = SequentialCost / ParallelismGain + ExchangeCost

ParallelismGain = MIN(WorkerCount, DOP)
  where DOP = degree of parallelism for operation

ExchangeCost = ScanExchangeCost + RepartitionExchangeCost
```

**Cost Heuristics**:
- Table scan parallelism = WorkerCount (perfect scaling up to I/O bottleneck)
- Join parallelism = 0.8 * WorkerCount (hash table overhead reduces efficiency)
- Aggregate parallelism = 0.9 * WorkerCount (local aggregation has good scaling)
- Sort parallelism = 0.7 * WorkerCount (merge overhead reduces efficiency)
- Exchange cost = data_volume * cost_factor (0.1 for repartition, 0.2 for broadcast)

**Threshold Logic**:
```
Skip parallelization if:
- Estimated output cardinality < 10,000 rows
- Query complexity is low (single table, no joins)
- Parallelism gain < 10% (communication overhead > benefit)
```

**Key Integration Points**:

3. `executor.Execute()` → recognize and execute parallel plans
4. `EXPLAIN` → show parallel plan structure

## Goals / Non-Goals

**Goals**:
- Enable automatic parallel plan generation for OLAP queries
- Achieve 3-8x speedup on modern CPUs (4-16 cores) for large queries
- Maintain correctness: parallel results = sequential results
- Minimize latency impact for small queries via threshold
- Support all join types (INNER, LEFT, RIGHT, FULL, CROSS)
- Work transparently without user involvement

**Non-Goals**:
- Distributed query execution (single machine only)
- Heterogeneous worker types (CPU affinity, NUMA aware)
- Dynamic plan reoptimization during execution
- Predicate pushdown optimization (handled separately)
- Query result caching (separate proposal)
- Adaptive parallelism (separate proposal)

## Decisions

**Decision 1: Morsel-based vs Range-based Partitioning**
- **Choice**: Morsel-based (chunk-based)
- **Rationale**: Simpler implementation, load-balanced, matches DuckDB approach
- **Alternative**: Range-based (predicate-based) - more complex, requires histogram

**Decision 2: Static vs Dynamic Parallelism**
- **Choice**: Static (decided at plan time)
- **Rationale**: Simpler implementation, deterministic, good for OLAP
- **Alternative**: Dynamic (decided at execution time) - more complex, better for OLTP

**Decision 3: Exchange Operator Implementation**
- **Choice**: Explicit operators (Exchange nodes in plan tree)
- **Rationale**: Visible in EXPLAIN, easier to debug and reason about
- **Alternative**: Implicit (handled at execution layer) - less visibility

**Decision 4: Cost Threshold Decisions**
- **Choice**: 10% parallelism gain threshold, 10k row cardinality threshold
- **Rationale**: Conservative, avoids parallel overhead for small queries
- **Alternative**: More aggressive (1% gain, 1k rows) - more parallelization, more overhead

## Risks / Trade-offs

**Risk 1: Over-parallelization of Small Queries**
- **Mitigation**: Cardinality threshold + cost-based threshold
- **Trade-off**: May skip profitable parallelization in rare cases

**Risk 2: Memory Overhead for Exchange Operators**
- **Mitigation**: Streaming exchange operators, limit broadcast to small data
- **Trade-off**: Repartition exchange may buffer large intermediate results

**Risk 3: Scheduler Contention**
- **Mitigation**: Fixed worker pool, no dynamic thread creation
- **Trade-off**: Cannot add more threads than configured pool size

**Risk 4: Incorrect Plans in Edge Cases**
- **Mitigation**: Comprehensive testing, race detector enabled, equivalence checking
- **Trade-off**: Testing complexity, potential latency in EXPLAIN ANALYZE

**Risk 5: Plan Size Explosion**
- **Mitigation**: Don't generate parallel plans for queries > 20 joins, use greedy heuristics
- **Trade-off**: Misses optimization opportunities for complex queries

## Open Questions

1. **Exchange Buffering**: Should repartition exchange buffer entire input before distribution, or stream in chunks?
   - Current plan: Stream in chunks with bounded memory

2. **Worker Count Selection**: Should adapt to CPU utilization or fixed to GOMAXPROCS?
   - Current plan: Fixed to GOMAXPROCS, with PRAGMA override option

3. **Broadcast Threshold**: What's the max size for broadcast exchange?
   - Current plan: 100MB (configurable via PRAGMA)

4. **Spill Management**: When repartition exchange exceeds memory, should it spill to temp storage?
   - Current plan: Yes, use temp directory with WAL-style recovery
